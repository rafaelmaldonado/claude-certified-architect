The agent loop, and the four ways to build one

Your win today: draw the agent loop from memory, and use the harness × deployment grid to pick between a manual loop, the Tool Runner, Managed Agents, and the Agent SDK. This is the beating heart of the 27% domain.

Recall from Lesson 1 You already decided whether to build an agent (the decision ladder). This lesson is the next question: once it is an agent, how do you build it? Same skill — tradeoffs — one level deeper.

1 · What an “agent” actually is

Strip away the hype and an agent is just a loop around the Messages API. The model is stateless, so something has to keep calling it, run the tools it asks for, feed results back, and decide when it’s done. That “something” is the loop.

The loop, in five beats

1. SEND    → call /v1/messages with the conversation + tool definitions
2. INSPECT → look at response.stop_reason
              • "end_turn"  → the model is done. EXIT the loop.
              • "tool_use"  → the model wants a tool. Continue ↓
3. EXECUTE → run each requested tool in your code
4. APPEND  → add the assistant turn (with tool_use blocks)
              + a user turn carrying the matching tool_result blocks
5. REPEAT  → go back to step 1 with the grown conversation
Two facts the exam checks (a) Every tool_result must carry the same tool_use_id as the request it answers. (b) The API is stateless — you resend the full, growing conversation every iteration. The server remembers nothing.

The exit conditions you must handle

stop_reason Meaning Loop action
end_turn Model finished naturally. Exit — you have the answer.
tool_use Model wants one or more tools. Run them, append results, continue.
max_tokens Hit the output cap mid-answer. Raise max_tokens / stream; don’t treat as done.
pause_turn A long server-side tool run paused. Re-send to resume — don’t add a “continue” message.
refusal Declined for safety. Surface it; don’t blindly retry.

Always cap the loop (e.g. max_iterations = 10) so a misbehaving model can’t spin forever — a reliability point that also lives in Domain 5.

2 · The mental model that unlocks the domain: harness × deployment

Two independent questions decide how you build any agent:

  • Who supplies the harness? (the loop + context management above)
  • Who supplies the deployment? (the infra it runs on)

These two axes are the whole game. Memorize this grid and most Domain-1 questions answer themselves:

Approach You write Harness Deployment Tools available
Manual loop
(API, your while)		 The whole loop You You host Only tools you define
Tool Runner
(client.beta.messages.tool_runner)		 Just the tool functions SDK You host Only tools you define
Managed Agents
(persisted agent + sessions)		 Agent config + tool results Anthropic Anthropic Hosted sandbox (bash/files/code) + MCP + your tools
Claude Agent SDK
(separate library)		 A prompt + options SDK (Claude Code) You host Built-in Read/Write/Edit/Bash/Grep/… + MCP + subagents
The one-sentence discriminator Manual loop, Tool Runner, and Agent SDK all leave deployment to you — they’re harness-only. Only Managed Agents also hosts the deployment (it runs the loop and a per-session sandbox). That single fact separates the four.

How to choose — the reasoning, not the memorization

  • Want to own every detail of the loop (custom control flow, no beta dependency)? → Manual loop.
  • Custom-tool agent, but don’t want to hand-write the loop? → Tool Runner (still gives you approval gates, retries, result tweaks via per-turn hooks — “I need control” is rarely a reason to drop to manual).
  • Want Anthropic to run the loop and host a workspace — persisted, versioned agents, long sessions, file mounts, scheduling? → Managed Agents.
  • Want a batteries-included coding/filesystem agent on your own infra? → Claude Agent SDK.
The trap that sinks people “Tool Runner” ≠ “Claude Agent SDK.” Tool Runner is a thin helper inside the regular API SDK — no built-in tools, no filesystem, you supply every tool. The Agent SDK is Claude Code as a library — ships built-in file/bash/search tools and the full harness. Both are harness-only. Expect at least one question probing exactly this.

3 · Worked example — the same task, two ways

Task: “What’s 2 + 3?” with an add tool. First, the manual loop (you own everything):

messages = [{"role": "user", "content": "What is 2 + 3?"}]
while True:
    resp = client.messages.create(model="claude-opus-4-8", max_tokens=1024,
                                  tools=tools, messages=messages)
    if resp.stop_reason == "end_turn":
        break                                   # ← exit condition
    messages.append({"role": "assistant", "content": resp.content})   # APPEND assistant
    results = []
    for block in resp.content:
        if block.type == "tool_use":
            out = run_tool(block.name, block.input)         # EXECUTE
            results.append({"type": "tool_result",
                            "tool_use_id": block.id,        # ← same id!
                            "content": out})
    messages.append({"role": "user", "content": results})  # APPEND results, REPEAT

Now the Tool Runner — the SDK runs those same five beats for you:

@beta_tool
def add(a: int, b: int) -> str:
    "Add two integers."
    return str(a + b)

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8", max_tokens=1024,
    tools=[add], messages=[{"role": "user", "content": "What is 2 + 3?"}])
for message in runner:      # loop, tool execution, and result feedback handled for you
    print(message)

Same loop, same result — you just chose who writes it. That choice is the exam skill.
